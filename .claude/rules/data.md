# Data Layout Rules

`data/` is the data layer of the repository. **Actual contents are git-untracked and managed by DVC + S3.** Only `*.dvc` / `*.md` / `*.gitkeep` files are tracked by git.

## Top-level layout (4 fixed layers)

Do **not** create any directory directly under `data/` other than the four layers below. When adding a new use case, classify it into one of these four layers and create a subdirectory under it.

```
data/
├── lake/        Raw data (collected as-is, no processing)
├── processed/   Pre-processed data (intermediate artifacts)
└── mart/        Curated data for training/evaluation (aggregated tables, etc.)
└── output/      Final pipeline artifacts
```

| Layer | Nature | Examples |
|---|---|---|
| `lake/` | Immutable, original. Re-acquisition required if lost | `lake/arc-agi-1/` (ARC-AGI-1 公開 train タスク JSON) / `lake/arc-gen-100k/` (ARC-GEN-100K 検証例) |
| `processed/` | Intermediate data derived from lake | 前処理済みのタスクテンソル（入力グリッド→`[1,10,30,30]` 等）のキャッシュ |
| `mart/` | Curated data read directly during training/evaluation | タスクごとの train/test/arc-gen ペアを学習用に整形したもの |
| `output/` | Pipeline execution results, regenerable | `output/onnx/` (タスクごとの `taskNNN.onnx`) / `output/submit/` (Kaggle 提出 `submission.zip` + 履歴) |

## How to choose a layer

- **Need re-acquisition if lost → `lake/`**
- **Reproducible by re-running code → `output/`**
- **Between lake and output, consumed by other pipelines → `processed/` or `mart/`**
- When in doubt: **inputs default to `lake/`, outputs default to `output/`**

## Subdirectory naming

- Directly under each layer: split by **use-case domain** (`arc-agi-1/`, `arc-gen-100k/`, `onnx/`, `submit/`)
- Below the domain: split by **task / approach** as needed (e.g. `onnx/task001/`)

## DVC management

> ✅ DVC を**正式採用**。`.dvc/` 初期化済み、S3 remote は Terraform（`infra/modules/dvc_remote`）が管理する専用バケット。ECS の自律ループ（生成 ONNX を `dvc push`）と submit タスク（`dvc pull`）の受け渡し経路でもある。詳細は [`infra/README.md`](../../infra/README.md)。

- lake / mart / output 配下を `dvc add` でディレクトリ単位に track し、生成された `*.dvc` のみ git commit（実体は git 無視）
- S3 remote（default `storage`）: `s3://neurogolf-dvc-<account_id>/remote`（バケットは Terraform で作成、`.dvc/config` の URL と一致）
- 認証: ローカルは `~/.aws` プロファイル、ECS は **task role の IAM**（`dvc[s3]` が boto3 経由で自動使用）
- 取得/送信: repo root で `uv --project backend run dvc pull` / `dvc push`

## .gitignore behavior

`.gitignore` ignores everything under `/data/**` **except**:

- `*.dvc` (DVC tracking files)
- `*.md` (README and similar)
- `.gitignore` (including those auto-generated by DVC)
- `.gitkeep` (placeholder for empty directories)

Real data (parquet, json.gz, pt, tar.gz, png, etc.) must never be committed.

## Worktree interaction

- In worktrees, the entire `data/` directory becomes a **symlink** to the main repository's `data/` (`dev/sync-data` sets this up)
- Real data generated or fetched inside a worktree must be merged into main and replaced with the symlink via `dev/sync-data`
- **Do not accumulate real data in a local `data/` inside a worktree.** Run `dev/sync-data` to merge as soon as data accumulates
- The DVC cache (`/Users/user/project/neurogolf/.dvc/cache`) is shared across all worktrees

## Prohibitions

- **No directories directly under `data/` other than lake / processed / mart / output** (e.g. legacy `data/matches/`, `data/submissions/`, `data/artifacts/` are abolished — re-classify them into the 4 layers)
- Never commit real data to git (manage via `.dvc` files)
- Do not create a nested symlink inside a worktree pointing back to main's `data/` (structures like `data/data` are forbidden)
- Use repository-root-relative paths (`data/...`), not hardcoded absolute paths (`/Users/.../data/...`)

parameter growth exceeded memory savings. A net-cost scan found `task173`, where the
full label image is large enough for crop-at-source to remain profitable.

## Hypothesis

The graph currently computes:

`1x1 Conv -> Cast -> Slice([0:25, 0:25])`

The full `30x30` label image is only used to create `grid`, the top-left `25x25` crop.
Replace the `1x1` label Conv with a no-padding sparse `6x6` Conv whose only non-zero
taps are at `(0,0)`. A `30x30` input then directly produces `25x25` output.

This removes the full float and uint8 label intermediates while adding only a small
sparse kernel.

## Method

1. Replace `task173_label_conv_w` from `(1,10,1,1)` to sparse `(1,10,6,6)`.
2. Route the Cast directly to `grid`.
3. Remove the dead Slice and unused crop constants if possible.
4. Audit only `task173.onnx` with `evaluate.audit_one`.
5. Submit and adopt only if Public Score improves.

## Risk

Low. This is a fixed top-left crop equivalence and does not depend on local examples.

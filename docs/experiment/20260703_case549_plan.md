## Background

Case548 increased parallelism but found that several high-gain near-miss
aliases deleted essential logic. Exact repairs for `task364`, `task324`,
`task154`, and `task343` restored the accepted baseline cost, while retained
gain variants stayed locally incorrect.

Therefore this cycle shifts from patching unsafe aliases to finding one task
where the accepted ONNX is structurally overbuilt and can be replaced by a
simpler exact formulation.

## Priority Rule

Use the metric directly:

```text
gain = ln(old_cost / new_cost)
```

Do not prioritize absolute cost alone. A standalone candidate must satisfy:

```text
n_fail = 0
forbidden ops = []
functions = 0
gain >= 0.020
```

A micro bundle can be submitted only if at most 5 current-baseline exact
non-risk candidates total at least `0.015`.

## Parallel Work

Three independent workers search for:

1. a new case3-style exact solver replacement for a non-risk task;
2. current-baseline cheaper replacements from `case2/golf_wins` and case3
   generated outputs;
3. non-stale exact candidates mined from recent `/private/tmp` summaries.

The main process owns only plan/result, gate review, full 400-task packaging,
Kaggle submit, and adoption/rejection.

## Submit Rule

Submit every 10-15 minute decision window only if a candidate or bundle passes
the gate. Every real submit must use a full 400-task directory built from the
accepted baseline, with only the intended task files overwritten.

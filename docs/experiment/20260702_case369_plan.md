# 20260702 case369 plan - medium-cost structural probe with submit-saving gate

## Context

Accepted Public Score is `7185.78` after case362. Cases 363-368 skipped Kaggle
submits because exact material was below the risk-adjusted thresholds or near
misses failed local correctness.

The competition metric is logarithmic:

```text
points = max(1, 25 - ln(cost))
cost = params + memory_bytes
gain = ln(old_cost / new_cost)
```

Therefore task priority is based on expected relative reduction, not absolute
cost. `n_fail=0` remains the hard gate.

Official Kaggle overview reconfirms the public rule: one ONNX per ARC-AGI public
training v1 task, points only for functionally correct networks, and cost is the
sum of parameters plus memory bytes.

## Target Selection

This cycle targets the remaining medium-cost memory-dominated tasks where a
branch, dtype boundary, or terminal renderer may be removable:

| task | current cost | reason to inspect | prior risk |
|---:|---:|---|---|
| 370 | `10393` | repeat candidate graph with several selector branches | case247 branch deletion failed; only dtype/terminal pruning is still plausible |
| 205 | `12494` | ArgMax/GridSample-heavy graph after the adopted AveragePool change | GridSample-to-Slice failed boundary cases in case311 |

| 340 | `10994` | many Cast/Slice/Concat/Equal nodes, possible guard aliases | case301 guard aliases were tiny or incorrect |

Primary probes are task370 and task205 because their cost is still memory
dominated and the prior logs leave narrow, different search surfaces. Task204
and task340 are fallback only if the primary probes produce no exact candidate.

## Method

1. Inspect current ONNX graph structure and initializers.
2. Try bounded scratch rewrites only under `/tmp`:
   - task370: terminal mask/dtype cleanup, single-consumer rewires, and only
     non-branch equivalents not covered by case247.
   - task205: boundary-safe simplifications around `GridSample`, dtype cleanup,
     and redundant reshape/cast/table removal.
3. Validate every candidate with `src.evaluate.audit_one` for the touched task
   only.
4. Apply to `data/output/onnx` only if `n_fail=0` and the candidate clears a
   submit gate or is intentionally being banked.

## Submit Gate

Use the existing submit-saving policy:

```text
low risk:    submit if gain >= 0.020, otherwise bank
medium risk: submit if gain >= 0.030, otherwise bank
high risk:   submit only if gain >= 0.080 and isolated
```

The 30-minute cadence is a maximum submit frequency, not a requirement to spend
a submit on weak material. Exact micro candidates below threshold are recorded
in `micro_backlog.md` instead.

## Expected Outcome

Best case is one exact medium-risk structural reduction with gain at least
`0.030`, submitted alone for attribution. If only tiny exact rewrites are found,
bank them and skip Kaggle submit.

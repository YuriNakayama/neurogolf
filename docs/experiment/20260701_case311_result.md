## Failure Mode

The `GridSample` path uses `padding_mode=zeros`. When the sampled 5x5 window
extends beyond the top or left edge, `GridSample` emits zeros for out-of-bounds
coordinates. A plain negative-step `Slice` truncates or changes shape instead.
ONNX Runtime warnings showed boundary outputs such as `[1,10,25,30]` or
`[1,10,30,25]` before final comparison failed.

Adding padding or gather-based clipping would require larger intermediates than
the current 1866-cost graph can afford, so this is not a viable improvement.

## Secondary Check

The same GridSample-to-Slice idea was briefly tested on `task205` because its
window is a positive-step 10x10 crop. It reduced local cost from `12494` to
`11976`, but failed `5` examples for the same boundary-padding reason. It is not
selected as a follow-up unless a cheaper boundary-zero construction is found.

## Conclusion

Rejected. `task400` looked like a high-ratio terminal-renderer win, but
`GridSample` zero-padding semantics are essential for boundary examples. Continue
searching for GridSample/Slice replacements only when the sampled window is
provably in bounds for all local and generated examples, or when boundary padding
can be implemented with less cost than the removed grid path.

# 20260703 case542 plan - scorable-only structural search after task135 regression

Accepted Public Score:

- `7187.41`

## Background

Case541 used one isolated submit on task135 because the candidate was exact and
small, but the accepted baseline was locally unscorable. Public Score regressed:

```text
7187.41 -> 7186.80
```

The result shows that unscorable-baseline replacements are not safe simply
because the candidate is exact on local public/arc-gen examples.

## Hypothesis

The next submit should only come from a task where both the accepted baseline
and candidate are scorable locally, so the relative gain is measurable before
spending a submit.

Large enough gains are unlikely to come from one-byte initializer edits. The
search should prioritize scorable non-risk tasks with structural redundancy:

- oversized dense kernels with many zero/duplicate slices, but transformed in a
  way the validator still scores;
- repeated constant branches where one branch can be removed without changing
  outputs;
- shape-preserving simplifications that reduce memory without introducing
  sparse initializers or negative-pad inference failures.

## Method

1. Exclude hard/review/runtime-risk tasks and known blocked families.
2. Exclude tasks whose accepted baseline has `cost=None`.
3. Search scorable non-risk tasks with cost `3000-30000` for structural
   reductions, not micro initializer dedupe.
4. Gate each candidate individually with `evaluate_candidate_gate`.
5. Submit only if standalone gain `>= 0.020` or a safe micro bundle reaches
   `0.015`.

## Submit Rule

Do not use `--skip-pre-submit-gate` in this cycle.

Submit only if the normal pre-submit gate returns `submit-bundle` or
`submit-micro-bundle`.

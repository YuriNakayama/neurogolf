# case229 result: task123 canvas index encoding

## 結果
- 対象: `data/output/onnx/task123.onnx`
- baseline: `cost=1343` (`params=940`, `memory=403`, `n_pass=265`, `n_fail=0`)
- 採否: 不採用
- Kaggle submit: なし

## 検証内容
`canvas_index (30,30)` は 900 要素の int32 initializer だが、値は `min(max(row, col), 10)` と一致する。相対削減率を狙って、小 initializer からの生成と dtype 変更を probe した。

| 候補 | cost | n_pass | n_fail | 判定 |
|---|---:|---:|---:|---|
| baseline | 1343 | 265 | 0 | ok |
| `canvas_index` int64 | 1343 | 265 | 0 | reject: cost unchanged |
| `canvas_index` int16/uint8/int8 | invalid | 0 | 0 | reject: `Gather` index type invalid |
| 11x11 core + `Pad` | 4169 | 265 | 0 | reject: memory 増で悪化 |
| row/col + `Max` + `Min` | 7704 | 265 | 0 | reject: memory 増で悪化 |
| row-only / col-only / 11x11 direct | invalid | 0 | 265 | reject: output shape mismatch |
| `Einsum(channel_weights)` -> `ArgMax` | 1353 | 265 | 0 | reject: params は減るが memory 増で悪化 |
| `channel_weights` float16 | invalid | 0 | 0 | reject: `Einsum` input dtype mismatch |

## 結論
`canvas_index` は構造的には圧縮可能だが、scorer は initializer を element 数、runtime tensor を bytes で数えるため、int32 の 30x30 index を動的生成すると memory が増えて baseline より高くなる。`Gather` index は int32/int64 が必要で、低 dtype 化もできない。

`n_fail=0` かつ cost 改善の候補がないため、ONNX は変更せず submit しない。

## 次の一手
大きな int32 index table は「計算で作る」より initializer のままが安い場合が多い。次は bool/uint8 の同 shape 中間を削れる候補、または one-node Conv/Einsum 系で重みの dtype/shape を保ったまま params を減らせる候補を優先する。

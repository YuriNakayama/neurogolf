## Main Probe

The target surface was the float16 content computation:

```text
eq_b -> Cast(float16) -> onehot16[1,9,10,10]
Einsum(onehot16, onehot16, color_values16, intervals16)
  -> h_content_3d/v_content_3d -> Cast(uint8)
```

The values are integer-valued, but ONNX Runtime blocks the narrower
representations:

| candidate | result | decision |
|---|---|---|
| `onehot16`, `intervals16`, `color_values16` all uint8 | `Einsum` kernel not implemented for uint8 | reject |
| feed bool `eq_b` directly to `Einsum` | invalid graph, `Einsum` rejects bool | reject |
| float32 control | exact but `8410 -> 10610` | reject |

The `intervals16` table is binary and sparse, but replacing it with a computed
interval mask would require extra 10x10x10-style intermediates and does not look
competitive with the current fp16 table.

# 20260630 case61 result — gravity タスク t078 を精査、多色 gravity は count-based floor が最適（提出なし）

## independent_impl（必須欄）
未試行角度「gravity/projection 系で floor が multi-op の高コストタスク」を全 floor から探索:
- gravity-solvable: **t032(down, floor910), t078(up, floor1293)** の2件。
- **t078 精査**: gravity-up（多色, 列ごとに非ゼロを上に詰める）266/266。
  - floor は **count-based**（ReduceL1 で列の非ゼロ数を数え配置）11node/1293。max 9個/列, **968列が多色**。
  - **多色 gravity は非線形**（色の順序保存 + gap 詰め）→ 単一 Conv 不可（t032 は構造単純で 910 で成立したが
    t078 は 9 深・多色で Conv 化できない）。
  - iterated MaxPool-shift（niter=10）で gravity を組むと per-cell 条件移動が多 op → 1293 超過見込み。

## 確定
- 多色 gravity(t078) は floor の **count-based 構成(1293)** が最適。単一 Conv も iterated shift も
  floor を下回れない。t032(910) は max3個・構造単純ゆえ単一 Conv が成立した特殊例。
- gravity/projection 系も floor が規則複雑度に応じ最適 op を選択済み（単純→Conv, 複雑→count）。

## 27サイクルの一貫知見
- floor は per-task で「規則複雑度に見合う最小 op 構成」に到達済み:
  単純局所→single-Conv(中間ゼロ), 多色gravity→count, flood-fill→uint8 MaxPool, denoise→矩形検出。
- 独自実装は各規則を正答構成できる（多数の novel solver を構築）が、floor の cost には届かない
  （parity が上限）。純増余地は本探索(27サイクル)で発見できず。

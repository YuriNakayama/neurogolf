# 20260701 case307 result - reject task005 ray assembly probe

## Result

No ONNX change and no submit.

`task005` was selected as a high-ratio candidate, but the pre-edit probe did not
find a safe structural rewrite with expected saving above the 2% threshold.

## Baseline

Current `task005.onnx`:

- params: 490
- memory: 6392
- cost: 6882
- `n_fail=0`
- 2% target: `< 6748`
- required saving: at least 138 cost

## Findings

The graph is visibly large:

- `Slice`: 101
- `Max`: 44
- `Concat`: 13

However, the memory is concentrated in a few semantic stages:

- color-index `Conv`: 900 bytes
- final `Pad`: 900 bytes
- `ray_total`: 529 bytes
- final stamp `QLinearConv`: 441 bytes
- input color crop/cast: 225 bytes
- six ray-row concat tensors: 69 bytes each

The repeated tiny `Slice`/`Max` nodes are mostly 3x3 or 7x7 ray pieces. Removing
one or two of them would not clear the 2% gate. Clearing the gate requires a
larger rewrite that removes `ray_total`, several ray rows, or the final stamp
stage. No exact, lower-memory replacement was identified.

Common-subexpression probing also found no duplicate `Slice` nodes with identical
dynamic inputs and initializer values.

## Conclusion

`task005` remains a plausible future semantic rewrite target, but not a good
single-cycle surgery target under the ratio gate. The current implementation is
large in node count, but its high-cost tensors correspond to necessary stencil
assembly stages.

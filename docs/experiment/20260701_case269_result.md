- 対象: `data/output/onnx/task300.onnx`

## 差分

winner channel から selector を作る部分を以下から:

```text
winner_channel -> Cast(float16) -> winner_f16
winner_f16 == channel_ids_f16 -> selector
selector -> Cast(float) -> selector_f  # [1,10,1,1]
```

以下へ変更した。

```text
winner_channel -> OneHot(depth=10, values=[0,1]) -> selector_f  # [1,10]
```

`winner_f16` と `channel_ids_f16` は後段の `patch` 作成で必要だったため残した。
`row_code` / `col_code` の `Einsum` は selector rank を下げて以下に変更した。

```text
bchw,bc,h->b
bchw,bc,w->b
```

## 結論

低 cost task でも、rank の過剰な selector 中間を削ると相対削減率で submit 価値が出る。
今回は memory が 10 bytes 減り、params が 3 増えたため net cost は 7 減だったが、Public LB にも反映された。

## 次の一手

`Equal -> Cast(float)` で one-hot selector を作り、実際には channel rank だけで足りる task を探す。
特に `cost < 1000` の task では、数 bytes から数十 bytes の net 削減でも 1% 以上になり得る。

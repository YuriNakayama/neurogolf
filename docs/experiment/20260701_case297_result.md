## Change

The p7 branch originally computed `p7_inv_wide [1,1,7,7]` and then sliced the
first spatial cell:

```text
MaxPool(inv30_f, pads=[1,1,4,4], strides=[1,1]) -> p7_inv_wide
Slice(p7_inv_wide, starts=[0,0], ends=[1,1], axes=[2,3]) -> p7_inv_cell
```

Since the kept cell is the first MaxPool output, changing the MaxPool stride to
`[99,99]` preserves the first pooling window while making the MaxPool emit only
`[1,1,1,1]`.  The MaxPool now writes `p7_inv_cell` directly and the Slice is
removed.

`p7_max_wide` was not changed because it has an additional full-grid consumer
(`p7_max_u8`).

## Conclusion

This is a small but proof-grade structural crop with no params increase.  It
confirmed that first-cell MaxPool witnesses can sometimes be source-cropped by
large strides.  The next search should look for the same pattern where the wide
MaxPool output has exactly one Slice consumer, and should skip non-first witness
cells unless the same window can be represented without negative padding.

# 20260701 case304 result - reject task280 ratio rewrite

## Result

No ONNX change and no submit.

`task280` was rejected at the pre-submit gate because no safe rewrite was found
that could clear the 2% ratio threshold.

## Baseline

Local audit / cost scan:

- params: 753
- memory: 5286
- cost: 6039
- required 2% target: `< 5919`
- preferred 5% target: `< 5752`

## Findings

The current graph is already a compact semantic renderer:

- find the two color-2 source cells with `Slice` + `TopK`
- infer direction and thickness from neighboring color-3 cells
- build separable row/column masks for the two generated objects
- render with one final `Einsum` directly into `output`

Main cost terms:

- `red_f` slice: 1296 bytes (`[1,1,18,18]` float32)
- final row/column term casts: 800 bytes total (`[5,20]` float32 x2)
- `spatial_select`: 600 params (`[20,30]` float32)
- first `TopK` index tensor: 288 bytes

The obvious alternatives are not safe or not beneficial:

- shrinking the 18x18 color-2 search range is not justified because all rows and
  columns 1..18 appear in the labeled examples;
- casting to float16 would require a large input-side cast before the final
  `Einsum`, which is more expensive than the saved intermediates;
- replacing the final separable renderer with a 20x20 core plus pad would create
  a large intermediate tensor;
- the generic case3 surgery pipeline produced no accepted improvement;
- removing the unused first output of the second `TopK` makes the ONNX graph
  invalid in ONNXRuntime.

## Conclusion

`task280` is not a good high-ratio target despite its medium cost and large node
count. The graph is over 100 nodes, but the expensive parts are tied to necessary
search/render structure. Continue with other medium-cost overbuilt candidates,
prioritizing tasks where the current graph is not already a separable semantic
solver.

# case208 result: task118 dtype memory trim

## 結果

- 対象: `task118`
- baseline: `cost=39237`, `params=4614`, `memory=34623`, `n_fail=0`
- submit: なし
- 採否: 不採用

## 試行

大きい dtype 変換を temp ONNX 上で alias し、scoreability と correctness を確認した。

| 変更 | status | 結果 |
|---|---|---|
| `input_u8 -> input` | session error | `QLinearConv` が float input を受けられない |
| `sel_pair -> sel_bool_pair` | session error | `Einsum` が bool input を受けられない |
| `red_pair16 -> red_pair` | session error | `Mul` で uint8 と float16 が混在 |
| `plus8 -> plus16` | session error | `Mul` で float16 と uint8 が混在 |
| `ng_patch_hash_bits -> ng_patch_hash_bits8` | session error | `MatMul` が uint8 input を受けられない |
| `ng_patch_match8 -> ng_patch_match_bool` | session error | `ScatterElements` の data/update dtype が不一致 |

## 結論

`task118` の大きな Cast は scorer memory 上は重いが、後続 op の dtype 制約に必要。単純 alias では ONNX として成立しないため、実 ONNX は変更していない。

## 次の一手

dtype alias は型制約で詰まりやすい。次は中 cost の未着手 graph から、branch 固定ではなく shape や Pad/Gather 周辺で同 dtype の alias ができる候補を探す。

# 20260701 case303 result - reject task270 semantic rewrite

## Result

No ONNX change and no submit.

`task270` was rejected at the pre-edit gate because the rule is real, but the
existing ONNX already implements the compact 1D semantic strategy. A rewrite is
unlikely to clear the new `old_cost / new_cost >= 1.02` threshold.

## Evidence

Local audit for current `task270.onnx`:

- params: 327
- memory: 2742
- cost: 3069
- `n_pass`: 266
- `n_fail`: 0

The semantic rule was checked against all available examples and matched all
266 pairs. However, graph inspection showed the current implementation is not a
broad lookup:

- row/column evidence is packed with `Einsum`
- center and marker colors are extracted with compact bit masks
- direction flags use `CumSum`, `ArgMax`, and `GatherElements`
- final composition is a single `Einsum` directly into `output`

The largest memory terms are already 1D:

- `row_basis`: 420 bytes
- `col_basis`: 420 bytes
- twelve shifted `Pad` tensors: 720 bytes total

A naive 2D mask/shift formulation would introduce 15x15 intermediates and is
expected to worsen cost. A smaller equivalent formulation was not identified.

## General answer to the improvement-width issue

Larger improvements require changing the task selection rule:

1. Score candidates by `ln(old_cost / estimated_new_cost)`, not by `old_cost`.
2. Do not spend cycles on `<1%` cleanup unless it is part of a larger same-task
   rewrite.
3. Prefer semantic replacements where the current graph is visibly generic:
   many `Gather`/`Reshape`/`Where` nodes, large selector initializers, repeated
   full-grid casts, or large temporary tensors.
4. Avoid tasks whose current graph is already a hand-golfed semantic solver, even
   if the absolute cost is nontrivial.
5. A good next search band is medium-cost overbuilt tasks where a simple ARC rule
   can replace a generated graph, for example tasks with cost 3000-10000 and
   unusually high node count.

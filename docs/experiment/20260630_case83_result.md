## 検証

### 入力 Cast

- graph input は float32 one-hot `[1,10,30,30]`。
- scorer も float32 を供給するため、graph input を uint8 にして `Cast(input)->v0` を消すことはできない。
- `Slice` を先に行ってから Cast すると、float Slice `[1,10,25,25]` が 25000B になり現行より悪化する。

### 末尾 bit propagation 短縮

`Concat n4676` の先頭列入力 `v4675` を一段前/別列の bitset へ置換し、dead node を剪定して監査。

| 置換 | nodes | cost | 結果 |
|---|---:|---:|---|
| `v4675 -> v4643` | 3016 | 47013 | `n_fail=215` |
| `v4675 -> v4611` | 3016 | 47013 | `n_fail=234` |
| `v4675 -> v4579` | 3016 | 47013 | `n_fail=235` |
| `v4675 -> v4547` | 3016 | 47013 | `n_fail=237` |
| `v4675 -> v4515` | 3016 | 47013 | `n_fail=244` |

末尾列の propagation は正答性に強く効いており、短縮不可。さらに scalar node を 20 個削っても cost は変わらず、削減対象としても効率が悪い。

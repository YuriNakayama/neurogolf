# 20260702 case446 result - regression/no-move guardrail

## Result

No Kaggle submit. This cycle investigated the recent regression/no-move pattern
and added a mechanical pre-submit guardrail.

## Findings

Recent failures came from four repeatable classes:

1. Local exactness was treated as hidden/Public safety.
   - case403 `task017` candidate-row pruning had local `n_fail=0` but sharply
     regressed Public.
   - case414 bundled locally exact micro candidates and regressed
     `7186.71 -> 7169.75`.
2. Standalone micros below `0.010` expected gain did not reliably move Public.
   - case411 `task342`, case412 `task338`, and case413 `task392` were exact but
     unchanged on Public.
3. Runtime-sensitive dtype/index rewrites produced platform risk.
   - `task233` and `task285` uint8 `TopK` families produced Kaggle ERROR.
   - Scatter/Gather index narrowing probes have repeatedly failed local/runtime
     compatibility.
4. Old scratch pools were stale.
   - cases438-445 re-gated many old candidates and found only local failures,
     no-gain exact rewrites, or bank-only micros.

The scoring formula still says to prioritize relative reduction:

```text
gain = ln(old_cost / new_cost)
```

but correctness and hidden/Public risk dominate. Numeric gain alone is not a
sufficient submit signal.

## Implemented Guardrail

Updated `backend/src/submit/gate.py`:

- `HARD_RISK_TASKS` now returns `blocked-known-risk` for known hidden/Public
  regressing families:
  - `task017`
  - `task101`
  - broad `task286`
  - contaminated `task305`
  - contaminated `task366`
- `REVIEW_RISK_TASKS` now returns `review-known-risk` for unisolated bundle
  members or unchanged standalone micros:
  - `task090`, `task209`, `task293`, `task308`, `task370`
  - `task338`, `task342`, `task392`
- `TOPK_RUNTIME_RISK_TASKS` now returns `review-known-risk` when the candidate
  still contains `TopK`:
  - `task233`
  - `task285`

This does not permanently ban all future work on these task numbers. It prevents
known-risk candidates from being promoted to `submit-candidate` solely because
they are local-exact and above the numeric gain threshold. A repaired candidate
now needs an explicit isolated review path.

## Remaining Gap

The actual `python -m submit submit` path still only performs structural
validation before packaging. It does not diff changed tasks against a baseline
and run `evaluate_candidate_gate` for each changed ONNX.

Next guardrail should be a bundle-level pre-submit gate:

1. compare candidate `--onnx-dir` against accepted baseline by task hash;
2. run `submit gate` only for changed tasks;
3. block submit if any changed task is `blocked-*`, `bank-low-gain`, or
   unapproved `review-*`;
4. record changed tasks, costs, gains, gate decisions, and Kaggle result in the
   submit history.

Until that exists, the operating rule remains: run `python -m submit gate` for
every changed task before any manual or scheduled submit.

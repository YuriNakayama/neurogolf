# case234 result: task202 color code dtype trim

## 結果
- 対象: `data/output/onnx/task202.onnx`
- baseline: `cost=11277` (`params=24`, `memory=11253`, `n_pass=230`, `n_fail=0`)
- 採否: 不採用
- Kaggle submit: なし

## 検証内容
冒頭の `Conv(input, cw) -> code_f` は色番号を float32 30x30 tensor として作るため、`ArgMax` や行/列方向の先行 reduction で置き換えられるか probe した。あわせて終端 `Where` branch の固定化も確認した。

| 候補 | cost | n_pass | n_fail | 判定 |
|---|---:|---:|---:|---|
| baseline | 11277 | 230 | 0 | ok |
| full `ArgMax(axis=1) -> uint8` | 15526 | 230 | 0 | reject: int64 中間で悪化 |
| full `ArgMax(axis=1) -> float` | 18467 | 230 | 0 | reject: memory 悪化 |
| row/col `ReduceMax(input)` then `ArgMax`, channel0 iszero | 13910 | 0 | 230 | reject |
| `bandc` true/false fixed | 10377 | 111-119 | 111-119 | reject |
| `hcell` true fixed | 9476 | 0 | 230 | reject |
| `hcell` false fixed | invalid | 0 | 0 | reject |
| `hb` true/false fixed | 10347 | 111-119 | 111-119 | reject |
| `cg0` true/false fixed | 5782-6716 | 0 | 230 | reject |
| `cg` true fixed | 9446 | 0 | 230 | reject |
| `cg` false fixed | invalid | 0 | 230 | reject |

## 結論
`Conv(cw)` は見た目は色番号化だが、`ArgMax` 置換では int64 中間が重くなり、完全一致しても cost が悪化した。行/列方向に先に畳む構成は `iszero` / code semantics が一致せず全ペア不一致だった。

終端 `Where` 固定は cost を大きく下げる候補もあるが、いずれも完全一致を保てない。`n_fail=0` かつ cost 改善の候補がないため、ONNX は変更せず submit しない。

## 次の一手
`ArgMax` は int64 中間が高く、float32 Conv を置き換えても悪化しやすい。次は `ArgMax` を使わず、既存 bool/uint8 mask の共有または複数 `Where` の exact prune を同一 task 内で束ねられる候補を探す。

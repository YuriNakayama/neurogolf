# 20260701 case306 result - reject task216 crop intermediate reduction

## Result

No ONNX change and no submit.

`task216` was rejected at the pre-edit gate because the dominant intermediates
are structurally necessary in the current compact crop solver, and the obvious
dtype/order changes are expected to worsen cost.

## Baseline

Current `task216.onnx`:

- params: 87
- memory: 9040
- cost: 9127
- 2% target: `< 8949`

## Findings

The largest tensors are:

- `c12_f32`: 3200 bytes, `[1,2,20,20]` float32
- `c12`: 800 bytes, `[1,2,20,20]` uint8
- final dynamic `crop`: 648 bytes, `[1,2,18,18]` uint8
- several 20x20 / 20x4 uint8 search tensors

The first slice extracts only colors 1 and 2 from the 20x20 input. The uint8 cast
is then reused by:

- 2x2 corner detection with `QLinearConv`
- row/column emptiness checks
- rectangle scoring
- final dynamic crop and pad into `output`

Removing the cast would force the downstream graph to operate on float32, which
adds more memory than it saves. Casting before the slice would materialize a
larger 10-channel grid and is also worse. Replacing the final crop with an
input-side float crop would similarly increase memory.

## Conclusion

No safe reduction above the 2% threshold was identified. The graph is memory
heavy, but the largest tensors are shared by the crop search and final output
path. Continue searching for a task where a large tensor is redundant rather than
central to the solver.

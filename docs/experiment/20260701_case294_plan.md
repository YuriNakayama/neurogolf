# case294 plan: task201 width crop-at-source Conv

## Background

Recent accepted cycles show that crop-at-source rewrites give materially larger deltas
than small aliases. `task048` was rejected as a candidate because the memory saved by
cropping a `30x1` tensor would be smaller than the extra sparse-kernel params. `task201`
has a better tradeoff: the full intermediate is `13x30`, then only `13x13` is used.

## Target

- Task: `task201`
- Current local cost: `4614`
- Expected new cost: about `3680`
- Expected ratio: about `1.25`

## Hypothesis

The graph currently computes:

`Conv -> Cast -> Slice([0:13, 0:13])`

The Conv output is `13x30`, but downstream only uses the first 13 columns. The original
Conv has kernel shape `(2,1)` with height dilation `17`, so height is already tight.
Replace it with a sparse `(2,18)` kernel, preserving the height dilation and putting the
old weights at width offset `0`. With no padding, the output width becomes `13`, exactly
matching the old left crop.

This removes the full-width float and uint8 intermediates while preserving the same
`labels` tensor consumed by the rest of the solver.

## Method

1. Replace `conv_weights` from `(1,10,2,1)` to sparse `(1,10,2,18)`.
2. Route the Cast directly to `labels`.
3. Remove the dead Slice and unused crop constants if possible.
4. Audit only `task201.onnx` with `evaluate.audit_one`.
5. Submit and adopt only if Public Score improves.

## Risk

Low. This is a fixed left-crop equivalence, independent of local example values.

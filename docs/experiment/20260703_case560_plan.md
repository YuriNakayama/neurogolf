## Background

Case559 increased subagent count, but no exact positive candidate was found. The
main bottleneck is now candidate supply, not Kaggle submission mechanics.

Several broad scans wasted time on runtime-invalid ONNX candidates or stale
scratch files. The next cycle should make candidate triage faster and more
interruptible so 15-minute cadence windows can either submit an exact candidate
or quickly record no-submit and move on.

## Hypothesis

A bounded current-baseline screening tool can improve throughput by:

1. comparing candidate files against the current accepted baseline;
2. running the existing `submit gate` logic per file;
3. continuing after runtime exceptions;
4. emitting exact positives and close misses in a machine-readable summary.

This does not change solver behavior or ONNX outputs by itself, but it should
increase the rate at which parallel workers produce usable candidates.

# 2026-06-30 case113 plan

## 方針変更

ユーザー再確認に従い、優先基準を「絶対 cost が大きいタスク」から「`old_cost / new_cost` が大きく見込めるタスク」へ変更する。

スコア改善は `ln(old_cost / new_cost)` なので、固定量の削減では小〜中 cost タスクの方が得点効率が高い。今後は以下を優先する。

1. `n_fail=0` を絶対条件にする。
2. 100B 未満の微小削減単体ではなく、相対削減率が数 % 以上になりうる候補を狙う。
3. 中〜高 cost で、1 個以上の 30x30/20x20 中間テンソルや dtype を削れる構造を優先する。

## 対象

- `task074.onnx`
- baseline: params 50, memory 9000, cost 9050, n_pass 265, n_fail 0
- graph:

```text
Conv -> Cast c0
Transpose c0 -> tT
Gather c0 -> tR
Max(c0, tT, tR) -> c1
Gather c1 -> tC
Max(c1, tC) -> c2
Equal(c2, chan) -> output
```

## 仮説

`task074` は symmetry / mirror 展開系の小ノード高 memory graph。最後の `Gather c1 -> tC` と `Max(c1, tC) -> c2` が冗長、または一部 arc-gen で不要なら、30x30 中間を複数削れて相対改善率が大きい。

## 手法

1. `Equal` 入力を `c2` から `c1` に差し替え、最後の column mirror 段を削除する。
2. `c1` の構成要素も 1 点ずつ検証する。
   - `c1 = Max(c0, tT, tR)` から `tR` を落とす。
   - `c1 = Max(c0, tT, tR)` から `tT` を落とす。
   - `c1` を `c0` に bypass する。
3. 各候補を prune し、`src.evaluate.scorer.audit_one` で `task074` の完全一致を確認する。
4. `n_fail=0` かつ cost が減る候補のみ実 ONNX に反映して submit する。

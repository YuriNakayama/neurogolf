# case242 result: task185 grid3 kernel crop

## 結果
- 対象: `data/output/onnx/task185.onnx`
- baseline: `cost=1961` (`params=310`, `memory=1651`, `n_pass=267`, `n_fail=0`)
- 採否: 不採用
- Kaggle submit: なし

## 検証内容
`grid3_kernel` `(1,10,4,4)` は非ゼロ bbox が `(row=2, col=2)` の 1x1 で、initializer params だけなら大きく削れる。ただし Conv は `stride=(3,3)` で、元の 4x4 kernel は入力位置 `2,5,8,...,26` を読んでいる。

一時 ONNX で以下を probe した。

| 候補 | cost | n_pass | n_fail | 判定 |
|---|---:|---:|---:|---|
| baseline | 1961 | 267 | 0 | ok |
| `grid3_kernel 4x4 -> 1x1`, 後段 Slice top-left | 2217 | 0 | 267 | reject |
| `grid3_kernel 4x4 -> 1x1`, 後段 Slice bottom-right | 2217 | 0 | 267 | reject |
| `grid3_kernel 4x4 -> 1x1`, Slice なし | 1817 | 0 | 267 | reject |

## 結論
kernel crop 自体は params を減らせるが、Conv の入力サンプリング位置が2セルずれるため完全一致しない。これを faithful に補正するには Conv 入力を `input[:,:,2:29,2:29]` のように事前 Slice する必要があるが、10 channel の大きな中間 tensor を追加して memory が大幅に悪化する見込み。`n_fail=0` かつ cost 改善の候補がないため、ONNX は変更せず submit しない。

## 次の一手
kernel crop は「非ゼロ bbox が左上原点に寄っている」または既存 pads で非負補正できるものを優先する。offset 補正に入力 Slice が必要な Conv は、小コスト task では memory 増が勝ちやすい。

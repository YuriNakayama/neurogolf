# 20260702 case419 result - regression guard hardening

## Summary

Implemented a mechanical pre-submit gate. No Kaggle submit.

## Changes

Added:

```text
uv --project backend run python -m submit gate CANDIDATE.onnx \
  --baseline-dir data/output/onnx \
  --task-dir data/lake/neurogolf-2026
```

The command extracts `taskNNN` from descriptive scratch names, compares the
candidate with the accepted baseline `taskNNN.onnx`, runs task-local
correctness, computes relative gain, checks ONNX functions and banned ops, and
prints a decision:

- `submit-candidate`
- `review-mid-gain`
- `bank-low-gain`
- `blocked-local-fail`
- `blocked-forbidden-op`
- other blocked states for baseline/no-gain/unscorable cases

## Decision

Adopt the guard. It does not replace Kaggle as the final judge, but it makes the
known failure modes explicit before spending a submit:

- high-gain but locally incorrect near misses are blocked;
- exact low-gain micros are banked instead of forced into cadence submits;
- mid-gain candidates require prior Public evidence or deliberate review;
- submit-candidate is reserved for isolated exact candidates with
  `gain >= 0.020`.

## Next Step

Use the new gate before staging every future candidate. Next search priority is
the safer family identified by the regression review: dense kernel or selector
table compression, repaired `n_fail=1` near misses, and same-task alias
extensions with prior Public evidence.

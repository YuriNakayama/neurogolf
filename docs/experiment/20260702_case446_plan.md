# 20260702 case446 plan - regression/no-move root cause guardrail

## Context

Accepted Public Score floor remains `7186.71`. Several recent cycles either
regressed or failed to move Public LB:

- case411 `task342`: isolated micro gain `0.00569`, Public unchanged;
- case412 `task338`: isolated micro gain `0.00671`, Public unchanged;
- case413 `task392`: isolated micro gain `0.00514`, Public unchanged;
- case414: five-task unisolated micro bundle, local `n_fail=0`, Public
  regressed `7186.71 -> 7169.75`.

The post-regression policy in `20260702_regression_analysis.md` is correct, but
part of it still depends on manual discipline. The one-task gate checks local
correctness and gain thresholds, but known Public-regressing and Kaggle-ERROR
families can still classify as submit-ready if they clear the numeric threshold.

## Hypothesis

The main causes are not the logarithmic scoring formula itself. They are:

1. treating local exactness as hidden/Public safety;
2. spending submit windows on standalone micros below practical movement
   threshold;
3. bundling unisolated micros to reach a numeric gate;
4. reusing stale scratch candidates against a changed accepted baseline;
5. not encoding known rejected families directly in the pre-submit gate.

Encoding the known-risk list in `submit gate` should reduce repeat regressions
without blocking normal fresh candidates. High-risk families should be blocked
or demoted to explicit review even when local `n_fail=0`.

## Method

1. Review recent experiment logs and subagent findings for concrete failure
   classes.
2. Update `backend/src/submit/gate.py` so known hidden-unsafe tasks,
   unchanged-micro tasks, and TopK runtime-risk tasks do not automatically
   become `submit-candidate`.
3. Add unit tests for the new gate outcomes.
4. Run focused tests and lint.
5. Record the remaining gap: the actual bundle submit path still needs a
   changed-task gate before packaging/submission.

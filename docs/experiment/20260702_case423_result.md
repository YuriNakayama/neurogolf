## task054 Inspection

Dominant tensors:

```text
label_f32:          float32 [1,1,30,30] = 3600 bytes, immediately Cast -> label_u8
fill_indices_4_8_4: int64   [4,8,4]     = 1024 bytes
v_indices_1_1_30_4: int64   [1,1,30,4]  =  960 bytes
```

`label_f32` is produced by `Conv(input, label_conv_w)` and cast to uint8. A
direct integer label path would need either an input-wide cast or a different
operator family, likely adding more memory than it removes. The index tensors
are plausible future targets, but changing `GatherND`/`ScatterND`/`ScatterElements`
index representation is runtime-sensitive and was not attempted in this cycle.

## task029 Inspection

Dominant tensors:

```text

col_selector: float32 [30,30] = 3600 bytes
```

These feed the final:

```text
Einsum(input, row_selector, col_selector)
```

Replacing the selectors with `Gather` would introduce a large float intermediate
such as `[1,10,30,30]`, which is larger than the saved selector memory. No
candidate was built.

## task044 Inspection


The graph already uses float16 after the crop. Avoiding `crop_f32` would require
casting the full input or replacing multiple downstream float16 paths. That is
not a narrow low-risk change.

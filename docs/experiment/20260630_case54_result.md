## independent_impl（必須欄・CumSum→MaxPool 置換で前進）
case53 の「CumSum は uint8 不可」を受け、t350 を **MaxPool だけ**で再構築:
- **規則を MaxPool 化**: 「整列ペア間」= 方向別 big-kernel MaxPool（kernel[H,1] pad非対称で
  上/下/左/右の presence）の AND/OR。**267/267 一致, nfail=0**。CumSum 不要。
- **uint8 MaxPool で cost 削減**: CumSum版 35185 → **MaxPool-uint8 版 20309**（1.7×減）。
- floor 構造を精読: floor は **26×24 crop**（max H=26,W=24）+ 単一 float slice(ch1) +
  free=1-ones 導出 + 同じ方向 MaxPool（hl/hr/hs/vu/vd/vs）。floor total **9012**。
- **floor 模倣を試行**: 26×24 crop + 単 slice → だが **out-of-grid マスク不足で nfail=265**
  （crop 内の grid 外セルが free=1 になり誤 fill）。presence マスク追加が必要（floor の `m`）。

## 現状の gap 分析
- 私の正答版(20309) vs floor(9012)。差の主因:
  1. **float slice 2個（ones_f+free_f=5408B）** vs floor 1個(2496B) → 単 slice 化で半減可能。
  2. **small [1,10,26,24] uint8（6240B）** → floor も同等だが私のは crop が 26×26 で大。
  3. presence マスク無しで 26×24 crop が破綻。floor は `m` マスクで grid 外を除外。
- これらを全て floor 通りにすれば 9012 に到達するが、**floor と同値で純増ならず**（parity）。

## 確定

  積み上げると parity に到達するが下回れない。**floor は op選択・crop・dtype すべて最適**。
- CumSum→MaxPool 置換は有効な技法（uint8 化可能に）。今後 projection/span 系に再利用。

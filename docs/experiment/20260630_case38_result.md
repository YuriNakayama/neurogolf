- **着眼**: scorer の `_calculate_params` は `sparse_initializer` と `Constant(sparse_value)` を
  **非ゼロ要素数のみ**で計上（src/evaluate/scorer.py:141-156）。floor の t171 Conv は
  weight [10,10,3,3]=900 のうち **非ゼロ 18 個のみ**。sparse 格納すれば params 900→18 の大幅減。
- **試行A: Constant(sparse_value)→Conv**:
  - params **910→28**（正答 nfail=0）。だが Constant が W中間テンソル [10,10,3,3]=3600B を生成し
    memory に加算 → **cost=3628 ≫ floor 910**。params 減を中間が相殺（case36 と同型）。
- **試行B: sparse_initializer を直接 Conv 入力に**:
  - ORT は sparse-init Conv を**ロード・実行可能**（中間ゼロ）。params=28。
  - だが **official `calculate_memory` の onnx.shape_inference が sparse op-input を拒否**:
    `InferenceError: (Conv): W has unsupported type: sparse_tensor(float)` → **memory=None=unscorable**。
  - さらに scorer の `_sanitize_model` は `graph.initializer` はリネームするが
    **`sparse_initializer` 名をリネームしない**バグがあり、ORT バインドも壊れる（safe_name 不一致）。

## 確定
- **sparse-tensor で params を削る独自実装は、official cost 関数（shape_inference）が
  sparse op-input を未サポートのため unscorable=0点**。params 910→28 は実現不可。
- Constant 経由は中間テンソルで相殺。直接 sparse-init は scorer 非対応。**どちらも floor 910 を
  下回れない**。floor の dense single-Conv（中間ゼロ, scorable）が依然下限。

## strategy.md 準拠の意義
- これは「不可能の一般化」ではなく、**sparse という具体的 op レベル新手法を実装・計測し、
  official scorer の制約で不可と実証**したもの。case30/35/36/37 に続く5本目の独自実装検証。

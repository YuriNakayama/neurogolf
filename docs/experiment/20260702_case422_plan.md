# 20260702 case422 plan - current-baseline cleanup scan

## Context

Accepted Public Score remains `7186.71`. Cases 420-421 ruled out the most
obvious residuals in dense kernel compression and old repaired near-miss
families.

The next useful action is to search from the current accepted ONNX files rather
than stale scratch directories. This lowers rediscovery and hidden-regression
risk.

## Hypothesis

There may still be low-risk exact improvements from:

- duplicate initializers with identical dtype/shape/value;
- unused initializers left after accepted rewrites;
- one-task cleanup candidates that do not alter selector/table coverage.

Most such wins will be small, but they should be banked accurately. If a single
task unexpectedly reaches `gain >= 0.020`, run the new `submit gate` and
consider isolated submit.

## Method

1. Scan current `data/output/onnx/task*.onnx` for duplicate and unused
   initializers.
2. Build candidate ONNX files by replacing duplicate initializer uses and
   deleting unused initializers within one task at a time.
3. Audit only changed candidate tasks with `submit gate`.
4. Submit only `decision=submit-candidate`; otherwise record banked/no-submit.

## Risk

This is lower risk than broad equivalence aliasing because it targets constants
that are byte-identical or graph-unreferenced. Still, every candidate must pass
task-local correctness before it can be banked.

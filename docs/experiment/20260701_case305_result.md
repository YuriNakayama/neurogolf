# 20260701 case305 result - reject task328 formula rewrite

## Result

No ONNX change and no submit.

`task328` has an exact coordinate formula, but the ONNX formula implementation is
too memory-heavy and does not beat the existing packed template-bank graph.

## Baseline

Current `task328.onnx`:

- params: 1940
- memory: 4964
- cost: 6904
- 2% target: `< 6769`

## Rule verified

A NumPy solver matched all 267 examples.

For each active corner, use Manhattan distance to assign each cell to its strict
nearest active corner; equal-distance cells remain blank. In local corner
coordinates `(a, b)`, a cell is painted when:

`(b <= a and a % 2 == 0) or (b >= a and b % 2 == 0)`

The painted color is the corresponding corner color.

## ONNX prototype

A formula graph was generated in `/tmp` with explicit static `value_info`:

- `n_pass`: 267
- `n_fail`: 0
- params: 1340
- memory: 36258
- cost: 37598

The rewrite removes much of the template-bank parameter count, but it creates
many `[1,18,18]` bool/int16 intermediates for distance, parity, territory, and
paint masks. Those intermediates dominate memory and make the model far worse.

## Conclusion

The existing packed `template_bank` implementation is superior for this scoring
metric. Even though it has a large initializer, it avoids materializing many
full 18x18 coordinate masks. Do not replace it with a direct coordinate formula
unless a much more fused representation is found.

## Next action

Move to a candidate where the expected reduction comes from removing one or two
large existing tensors rather than replacing a compact lookup with many masks.
`task216` is the next candidate because its current cost is 9127 and the scan
reported large crop intermediates.

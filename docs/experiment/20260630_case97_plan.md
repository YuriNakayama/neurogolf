# 20260630 case97 plan — t002 の bit-trick flood 後半反復短縮を検証する

## 背景

- 現 Public best は **7180.89**。
- `task002.onnx` は faithful baseline で cost **21089**（params 209, memory 20880）。
- 過去ログでは t002 は enclosed-fill で、現 floor は uint8 flood ではなく bit-manipulation graph。157 node のうち後半に `n_x0..n_x9` の近傍拡張反復がある。
- 手構築 flood-fill は cost 面で棄却済み。今回の狙いは現 graph の反復余剰や見かけ上の重複を一点検証することに限定する。

## 仮説

`n_x9` までの後半 10反復に余剰があれば、`out_bits` の入力を `n_x8` / `n_x7` へ置換して 1反復あたり複数の `BitShift/Gather/BitwiseOr/And` 中間を削れる。また `n_lk` / `n_rk` は同じ `BitShift(..., shift1)` に見えるため、左右方向が実際に同一なら片方を削れる可能性がある。

## 対象タスク

- `task002`

## 手法

1. ORT profile で memory 上位 tensor を確認する。
2. `/tmp` variant で以下を試す。
   - `out_bits` 入力 `n_x9 -> n_x8`
   - `n_x9 -> n_x7`
   - `n_x9 -> n_x6`
   - `n_lk` / `n_rk` の片側削除または `n_lrk -> n_lk` / `n_rk`
3. faithful `audit_one` が `n_fail=0` の候補だけ cost を比較する。

## 採否

- faithful `audit_one` が `n_fail=0` かつ cost が 21089 未満なら `task002.onnx` を差し替えて submit する。
- Public Score が 7180.89 を上回れば採用し、DVC push と git commit/push を行う。
- 壊れる候補、または cost が変わらない候補は破棄する。

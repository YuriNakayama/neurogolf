# case291 plan: task187 dual crop-at-source Conv

## Background

case290 showed that structural crop-at-source rewrites produce a larger and safer gain
than tiny local aliases. For larger per-cycle improvements, this cycle targets two
full-size intermediates in the same task rather than only one.

## Target

- Task: `task187`
- Current local cost: `35564`
- Strategy: remove two `30x30 -> 25x25` materialization paths.

## Hypothesis

The graph currently computes full `30x30` tensors and then slices the top-left `25x25`
region:

1. `1x1 Conv -> Cast -> Slice` for `label_u8_25`
2. `ReduceSum(channel) -> Cast -> Slice` for `grid_b_25`

Both crops can be computed directly with sparse `6x6` Convs whose only active taps are
at the top-left kernel location. With a `30x30` input and no padding, a `6x6` Conv
produces `25x25` output.

This is a shape/operator equivalence:

- sparse label Conv equals the original full label Conv restricted to `[0:25, 0:25]`
- sparse occupancy Conv equals the original channel ReduceSum restricted to
  `[0:25, 0:25]`

The rewrite does not rely on local example coincidences.

## Method

1. Replace `label_conv_w` with sparse `(1,10,6,6)` weights and route Cast directly to
   `label_u8_25`.
2. Replace `ReduceSum(channel)` plus full Cast/Slice with a sparse occupancy Conv and
   route Cast directly to `grid_b_25`.
3. Remove dead Slice nodes/constants if unused.
4. Audit only `task187.onnx` with `evaluate.audit_one`.
5. Submit and adopt only if Kaggle Public Score improves.

## Priority Rule

This case deliberately targets multiple full-size intermediates in one task. That is the
main way to raise per-cycle delta: combine same-proof structural deletions inside a
single task while keeping the one-task submission discipline.

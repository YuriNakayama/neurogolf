# 20260630 case49 result — flood-fill を uint8+crop で floor の1.38倍まで最適化（t251 6798 vs floor4920, 提出なし）

## independent_impl（必須欄・大幅前進、floor の符号化を解明）
floor t251 net を精読し、その最適化技法を独自実装に取り込んだ:

### floor の技法（解明）
- crop [1,*,12,12] + **全中間 uint8（1byte）** + 8×(MaxPool+Mul) + 3ch処理 + channel-pad。
- 種は precomputed `frame`（border mask initializer）を直接使用（dilation 不要）。
- opset **17**（uint8 Sub/Mul 可）。

### 独自実装の cost 削減積み上げ（t251, floor=4920）
| 版 | cost | 倍率 | 鍵 |
|---|---|---|---|
| naive 30×30 float | 111k | 23× | — |
| crop 12×12 | 34k | 6.9× | crop技法 |
| 単channel | 14.6k | 3.0× | channel削減 |
| **uint8 中間** | 8.1k | 1.65× | **dtype 1byte化（4×削減）** |
| **uint8 Concat** | **6798** | **1.38×** | float cast 除去 |

### 残差（floor 4920 まで -1878）
- 末尾の `small3f` float [1,3,12,12]=1728B cast が主因。floor は Concat(uint8)→Pad で float cast を
  さらに圧縮。あと一歩。MaxPool 数(8)・iteration は floor と同等。

## 前進の意義（以前の結論を大きく更新）
- 「独自実装は中間で常に大きく負ける」は**誤り**だった。crop+単channel+**uint8 中間**+frame seed の
  技法積み上げで、naive 比 **16倍** cost 削減し floor の 1.38倍まで到達。
- **uint8 中間（opset17）が決定的**: memory=Σ(n×itemsize) で dtype を 4byte→1byte にすると 4×削減。
  これは私が見落としていた最大のレバー。flood-fill 系・多ノード float net 系に広く適用可能。

excluding dtype-style reductions reduces Kaggle ERROR risk, but the candidate

Allowing narrow, low-risk dtype/initializer rewrites on non-runtime-sensitive
tasks can restore candidate supply without repeating the `task316` error.

Avoid:

- `task316` and all review/hard/runtime risk tasks in `submit.gate`;
- index-heavy rewrites around `GatherND`, `ScatterND`, `ScatterElements`;
- `TopK` score dtype changes;
- broad alias bundles with local hidden-safety concerns.

Prefer:

- duplicate/unused initializer cleanup;
- scalar/shape initializer compression;
- dtype narrowing only for constants feeding non-index arithmetic/comparison ops;
- one task per submit candidate.

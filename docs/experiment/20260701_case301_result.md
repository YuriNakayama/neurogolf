### Local Structural Scans

Checked high-visibility patterns manually:

- `task064` cast-before-slice:
  moving `Cast` after `Slice` would add cropped float32/int64 tensors and can
  cost more than the bool/uint8 intermediates it removes.
- `task338` repeated zero channel in final `Concat`:
  the repeated `zero_b` tensor is shared, so the apparent repeated-channel
  saving is not real scorer memory.  Replacing it with a tensor initializer or
  generated zero plane costs about the same.
- `task340` shared `idx_u8`:
  removing the full `1x1 Conv -> Cast` would require regenerating row, column,
  and center crops with larger kernels or wider input slices.  Net cost is not
  favorable.
- `task110` remaining wide label / MaxPool tensors:
  the wide producers still feed full-grid consumers, so the remaining single
  cells cannot remove the producer.

## Next Step

Move from static graph cleanup to deeper task-specific semantic search:

1. choose one medium-cost task where current graph has a real algorithmic
   bottleneck, not just local aliases;
2. derive the ARC rule from examples;
3. design a smaller ONNX solver or a structural rewrite with expected
   `old_cost / new_cost >= 1.02`;
4. only then write the next implementation plan and submit.

Good starting families remain terminal dtype/final-op rewrites and source-crop
rewrites, but only when the net scored cost accounts for added params and newly
introduced intermediate tensors.

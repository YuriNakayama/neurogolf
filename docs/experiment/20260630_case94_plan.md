# 20260630 case94 plan — t364 の A/B propagation 反復短縮を検証する

## 背景

- 現 Public best は **7180.87**。
- `task364.onnx` は faithful baseline で cost **23013**（params 113, memory 22900）。
- Graph は `gA` / `gB` から `MaxPool -> Mul(Gu)` を 7 段ずつ展開し、`PA7 * PB7` の値で recolor 種別を決める。
- 過去ログでは flood-fill 系として反復が重いとされているが、現在 floor に対する片側・最終段の局所短縮を改めて一点検証する。

## 仮説

7段反復の最終 `MaxPool` が余剰なら、`PA7` または `PB7` を `SA6` / `SB6` へ置換して 1段分の 20x20 中間を削れる。片側だけ余剰の可能性もある。

## 対象タスク

- `task364`

## 手法

1. ORT profile で memory 上位 tensor を確認する。
2. `/tmp` variant で以下を試す。
   - `PA7 -> SA6`
   - `PB7 -> SB6`
   - `PA7 -> SA6` かつ `PB7 -> SB6`
   - さらに 6段未満への短縮
   - `value_info` 全削除
3. faithful `audit_one` が `n_fail=0` の候補だけ cost を比較する。

## 採否

- faithful `audit_one` が `n_fail=0` かつ cost が 23013 未満なら `task364.onnx` を差し替えて submit する。
- Public Score が 7180.87 を上回れば採用し、DVC push と git commit/push を行う。
- 壊れる候補、または cost が変わらない候補は破棄する。

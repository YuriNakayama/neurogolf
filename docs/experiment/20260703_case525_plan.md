# 20260703 case525 plan - parallel submit-cadence recovery

## Background

Latest accepted Public Score remains 7187.37 at case517. Cases 518-524 did not
submit because no candidate passed the local gate:

- high-gain aliases mostly had `n_fail > 0`
- exact hits were either below standalone threshold or on risky bundle families
- broad worker ranges were too slow for the requested 15 minute submit cadence

The scoring metric is logarithmic:

```text
gain = ln(old_cost / new_cost)
```

So priority is not absolute cost. Priority is exact candidates with large
relative reduction, while also banking exact micro gains for later bundles.

## Hypothesis

Submit cadence can be recovered by splitting work into independent fast lanes:

1. re-gate known curated `golf_wins` against current baselines to find stale
   slack already present in the repo
2. mine accepted cases for reusable transformation families instead of repeating
   broad same-shape alias scans
3. run a narrow low-risk exact search and bank all exact micro candidates

This should reduce wasted cycles from local-fail near misses and increase the
chance of a safe submit every ~15 minutes when a candidate exists.

## Target

Primary target is any non-risk task with:

- `n_fail = 0`
- no forbidden ops/functions
- `gain >= 0.020` for standalone submit, or exact micro candidate to bank
- no recent blocked-family evidence unless isolated and repaired

Excluded recent blocked targets for this case:

```text
task011, task033, task157, task243
```

## Method

Parallel workers:

- worker A: evaluate `backend/pipeline/case2/golf_wins/task*.onnx` against
  current `data/output/onnx` baselines
- worker B: mine `docs/experiment` and `git log` for accepted families since
  case450 and list concrete reusable targets
- worker C: narrow bounded search on low-risk medium-cost tasks, banking exact
  micro gains and reporting best near misses separately

Parent process:

- keep a case525 result log
- submit immediately when a standalone `submit-candidate` appears
- submit a micro bundle only when all members are exact, low-risk, and total
  gain clears the configured bundle gate
- if no safe candidate appears, record the no-submit reason and keep the banked
  candidate list for the next cycle

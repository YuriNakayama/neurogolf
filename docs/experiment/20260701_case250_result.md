## 試行
case249 の横展開として、4つの bool mask Cast を float32 から float16 に変更し、`z` に `Cast(float16)` を追加して MatMul 入力型を揃えた。

対象:

```text
left_sum16   = MatMul(z, left_colsR)
right_sum16  = MatMul(z, right_colsR)
top_sum16    = MatMul(top_rowsR, z)
bottom_sum16 = MatMul(bottom_rowsR, z)
```

## 足切り
`/tmp/task255_f16_matmul.onnx` の `audit_one`:

```text
status=session_error:
Type Error: Type (tensor(float)) of output arg (...) does not match expected type (tensor(float16)).
```

下流が float32 のまま推論される箇所と合わず、ONNX Runtime が session を作れなかった。さらに、仮に全下流を float16 化できても、`z` は `[1,1,30,30]` のため `z_f16` 中間が 1800 bytes 増え、mask/output の削減を食う見込み。

## 結論
不採用。

case249 の float16 finalization は、final 演算が input から切り離されている場合に有効だった。`task255` は input slice `z` が直接 MatMul に入るため、同じ横展開では成立しない。ONNX 本体は変更しない。

## 次の一手
dtype 横展開は「final 演算が input を直接含まず、全入力を小さい float16 中間または initializer にできる」task に限定する。次は未言及 mid-cost の `task310` / `task091` / `task330` / `task070` などで、terminal mask・branch・dtype 以外の構造削減を探す。

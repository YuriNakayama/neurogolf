# case228 result: task147 scatter index trim

## 結果
- 対象: `data/output/onnx/task147.onnx`
- baseline: `cost=532` (`params=100`, `memory=432`, `n_pass=267`, `n_fail=0`)
- 採否: 不採用
- Kaggle submit: なし

## 検証内容
`scatter_indices` は `(1,2,6,6)` だが、channel 0 は全要素 3、channel 1 は全要素 8 だった。相対削減率を狙って index initializer の縮小と Conv output channel の削減を probe した。

| 候補 | cost | n_pass | n_fail | 判定 |
|---|---:|---:|---:|---|
| baseline | 532 | 267 | 0 | ok |
| `scatter_indices -> (1,2,1,1)` | 462 | 0 | 267 | reject |
| `scatter_indices -> (2,1,1)` | 462 | 0 | 267 | reject |
| `scatter_indices -> (2,)` | 462 | 0 | 267 | reject |
| `scatter_indices -> scalar 3/8` | 461 | 0 | 267 | reject |
| all indices 3/8 | 532 | 0 | 267 | reject |
| zero/duplicate Conv ch0/ch1 | 532 | 0 | 267 | reject |
| Conv+scatter 1 channel ch0/ch1 | 342 | 0 | 267 | reject |

## 結論
`ScatterElements` は `indices` と `updates` の shape を実質一致させる必要があり、broadcast 形状への縮小は全ペア不一致になった。2 channel の更新も両方必要で、片側削減は成立しなかった。

`n_fail=0` かつ cost 改善の候補がないため、ONNX は変更せず submit しない。

## 次の一手
小型タスクでも相対削減率が大きい候補を優先する。特に低〜中 cost で大きな定数テーブル、重複 initializer、Cast/Reshape/Slice の固定形状が残るタスクを次に確認する。

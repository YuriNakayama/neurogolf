# 20260702 case419 plan - regression guard hardening

## Context

Accepted Public Score remains `7186.71`. Recent cycles produced unchanged
submits and one sharp regression:

- case411-413: standalone low-gain exact micros did not move Public Score;
- case414: five-task locally exact micro bundle regressed Public Score;
- case418: high-gain `task349`/`task364` near misses were locally incorrect.

The main failure mode is not the logarithmic cost formula. It is submit
selection: locally exact candidates were treated as submit-worthy even when
they were below movement threshold, unisolated, or from a hidden-sensitive
family.

## Hypothesis

Adding a mechanical pre-submit gate will reduce wasted submits and prevent
known weak candidates from reaching Kaggle. The gate should compare one
candidate ONNX against the accepted baseline, run task-local correctness,
compute relative gain, and classify the candidate before staging.

## Implementation Plan

Add a `submit gate` CLI that reports:

```text
baseline cost
candidate cost
gain = ln(baseline / candidate)
status / n_fail
function count
forbidden ops
decision
reason
```

Decision policy:

- local fail, functions, forbidden ops, no-gain: block;
- `gain >= 0.020`: submit candidate if mechanically explainable;
- `0.010 <= gain < 0.020`: review only with prior Kaggle-valid evidence;
- `gain < 0.010`: bank only.

## Validation

Use unit tests for the decision thresholds and run the CLI on a known
case418-local-fail candidate to verify it blocks before submit.

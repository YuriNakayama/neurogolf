# 20260702 case381 plan - unary equivalence bundle scan

## Context

Accepted Public Score is `7185.88`.

Case380 found many exact Clip bypass micros, but the best same-task bundle was
only:

```text
task076: 18743 -> 18683, gain 0.003206
```

This is too small for a standalone submit. The submit budget should be spent
only when the expected movement is large enough, while exact micros are banked.

The contest score is logarithmic:

```text
points = max(1, 25 - ln(cost))
cost = params + memory_bytes
gain = ln(old_cost / new_cost)
```

So the target is relative cost reduction, not high absolute cost.

## Hypothesis

Some current ONNX graphs still contain one-input nodes whose output is
equivalent to the input for all benchmark cases, or whose downstream consumers
only need the weaker representation. Removing a single `Cast`, `Where`,
`Expand`, `Squeeze`, `Unsqueeze`, `Reshape`, `Abs`, `Neg`, `Not`, or similar
unary value path may expose a compatible same-task bundle.

Single rewires may be too small. The useful target is a compatible bundle in one
task that clears the low-risk gate.

## Method

1. Scan current `data/output/onnx/task*.onnx` for one-input/single-output nodes.
2. For each candidate node, rewire consumers of the output to the data input,
   remove stale graph metadata, and run `evaluate.audit_one` on that task only.
3. Keep exact candidates with `n_fail=0` and lower cost.
4. For tasks with multiple exact wins, greedily test compatible bundles.

Exclude known unsafe or stale families:

- `task002` bit-topology local positives
- `task118` table pruning
- `task233` uint8 `TopK` value-path compression
- broad `task286` local-equivalence bundles
- Clip-only micros already banked in case380 unless they combine with a fresh
  non-Clip reduction

## Gate

Submit only if a fresh low-risk candidate clears:

```text
gain >= 0.020
```

Otherwise record exact candidates in the result and add useful micros to the
backlog for later batching.

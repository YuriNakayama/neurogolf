## Hypothesis

The most useful next candidate is not another broad local-equivalence rewrite.
The recent failures show that large local-only semantic rewrites can regress
hidden/Public validation. The next cycle should search for one of:

1. A non-risk standalone exact candidate with relative gain `>=0.020`.
2. A same-task compatible stack on a known safe task that clears `>=0.020`.
3. A low-risk exact bundle of at most five tasks with total gain `>=0.015`.

Cost-only candidates are useful only as a filter; every candidate must pass
fresh full correctness for the touched task before any submit.

## Method

1. Re-gate fresh backlog and recent scratch candidates against the current
   accepted baseline.
2. Exclude hard/review/runtime-risk tasks unless the candidate is isolated and
   explicitly reviewed.
3. Prioritize relative gain (`ln(old_cost / new_cost)`) over absolute cost.
4. Use individual `submit gate` / `audit_one` only for changed tasks.
5. If submitting, build a full 400-task directory first:

```text
cp data/output/onnx/task*.onnx FULL_DIR/
cp candidate_taskNNN.onnx FULL_DIR/taskNNN.onnx
```

Then run dry-run and real submit from `FULL_DIR`.

## Submit Rule

Submit only if:

- standalone gain `>=0.020`, or
- micro bundle has at most five changed non-risk tasks and total gain `>=0.015`.

Never submit a one-file directory for an ordinary improvement cycle.
